# 评审规则对齐说明

## 一、评分结构

| 评分项 | 权重 | 本项目对应设计 |
|---|---:|---|
| 作品完整度与创新性 | 40% | Multi-Agent、Story Graph、Character Memory、YAML Schema、自动分镜、质量评估 |
| 开发过程与质量 | 40% | Git/PR规范、模块化架构、测试用例、可运行 Demo、依赖说明 |
| 演示与表达 | 20% | Demo 视频脚本、README、流程图、示例输入输出 |

## 二、作品完整度与创新性

### 1. 基础功能完整

项目满足题目核心要求：

- 支持 3 个章节以上小说输入；
- 自动解析人物、场景、事件、对白；
- 自动生成结构化剧本；
- 输出 YAML 格式；
- 提供 YAML Schema 说明文档；
- 说明 Schema 的设计原因。

### 2. 创新点设计

| 创新点 | 说明 |
|---|---|
| Multi-Agent 协同改编 | 将小说解析、角色建模、剧情规划、对白生成、质量审查拆分为多个智能体 |
| Story Graph 剧情图谱 | 将人物、事件、地点、冲突建立图结构，保证长篇剧情连贯 |
| Character Memory 角色记忆 | 存储人物性格、目标、关系与成长线，减少 AI 遗忘设定 |
| Source Trace 来源追踪 | 每个场景和对白可以追溯到原小说章节与段落 |
| 自动分镜建议 | 在剧本 YAML 中附带镜头类型、景别、画面重点 |
| AI 质量评估 | 自动评估剧情连贯性、人物一致性、对白自然度和场景转场 |

## 三、开发过程与质量

项目建议采用 PR 驱动开发：

```text
main
├── feature/novel-parser
├── feature/yaml-schema
├── feature/script-generator
├── feature/story-graph
├── feature/reviewer
└── feature/frontend-demo
```

每个 PR 只做一件事，并说明：

1. 新增/修改内容；
2. 技术实现思路；
3. 测试方式；
4. 截图或运行结果。

## 四、演示与表达

Demo 视频建议控制在 3-5 分钟：

1. 介绍项目背景；
2. 展示小说输入；
3. 展示 AI 分析过程；
4. 展示 YAML 剧本输出；
5. 展示 Schema 文档；
6. 展示质量评估；
7. 总结创新点。
